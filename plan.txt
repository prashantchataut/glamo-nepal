### Fix 1: Wishlist sync error

**Root cause:** `src/store/useWishlistStore.ts` line 102 — `response.data` is an object `{ items: [...] }` but code expects it to be the array directly. The `.map()` call on line 105 fails because objects don't have `.map()`. The error is caught by try/catch on line 126, which displays "Wishlist sync failed" toast.

**Fix:**
1. In `src/store/useWishlistStore.ts` line 102, change:
   ```typescript
   const serverItems = response.data ?? [];
   ```
   to:
   ```typescript
   const serverItems = (response.data as { items: WishlistItem[] })?.items ?? [];
   ```
2. Fix the type parameter on `wishlistApi.list()` in `src/lib/api/admin.ts` (or wherever the wishlist API is typed) to reflect the actual response shape `{ items: WishlistItem[] }` instead of `WishlistItem[]`.
3. Also check if `wishlistItemToProduct` on line 105 expects items or the wrapper object.

### Fix 2: cod_fee SQL error on order placement

**Root cause:** Migration `backend/migrations/0003_add_order_fee_columns.sql` adds `cod_fee` column to the `orders` table but this migration was never run against the production database. The `INSERT` in `backend/src/modules/orders/order.service.ts` line 428 includes `cod_fee` in the column list.

**Fix:**
1. Run migration 0003 against the Turso database, OR
2. If the schema.sql already has `cod_fee` (check `backend/src/config/schema.sql` line 295), re-initialize from schema.sql
3. Verify the column exists: `PRAGMA table_info(orders)` should show `cod_fee`
4. Run the migration script: check `backend/src/scripts/migrate.ts` for how to run migrations
5. If using Turso CLI: read the migration file and apply it

### Fix 3: Product page "(l ?? []).map is not a function"

**Root cause:** Same root cause as Fix 1. The wishlist sync is triggered on page load via `FirebaseAuthProvider` auth state change → `useWishlistStore.getState().syncFromServer()` → crashes on the same `.map()` call. The minified variable `l` = `response.data`.

**Fix:** This is automatically fixed when Fix 1 is applied, since the wishlist sync no longer throws.

### Fix 4: Settings page won't save

**Root cause:** Two independent blockers:

**Blocker A — Payload format mismatch:**
- Frontend (`src/lib/api/admin.ts` lines 874-878) sends: `Record<string, string>` = `{ "site_name": "Glamo Nepal", "cod_enabled": "true", ... }`
- Backend (`backend/src/modules/settings/settings.schema.ts` lines 3-8) expects: `{ settings: [{ key: "site_name", value: "Glamo Nepal" }, ...] }`
- Result: Zod validation always fails with 400.

**Fix A:** Change the frontend to send the correct format:
```typescript
// In src/lib/api/admin.ts, the saveSettings function
// Change from: body: JSON.stringify(data) where data is Record<string, string>
// To:
const settingsArray = Object.entries(data).map(([key, value]) => ({ key, value }));
body: JSON.stringify({ settings: settingsArray })
```

**Blocker B — Role requirement:**
- Backend route (`backend/src/modules/settings/settings.routes.ts` line 17) requires `['SUPER_ADMIN']`
- If the logged-in admin is only `ADMIN`, they get 403

**Fix B:** Change the role requirement to `['ADMIN']` in `backend/src/modules/settings/settings.routes.ts`:
```typescript
.put(requireRole(['ADMIN']), validateBody(updateSettingsSchema), updateSettings)
```

### Fix 5: Coupons not showing after creation

**Root cause:** RBAC gated to `SUPER_ADMIN` only:
- Frontend sidebar (`src/components/admin/AdminSidebar.tsx` line 77): `coupons: "SUPER_ADMIN"`
- Backend routes (`backend/src/modules/coupons/coupon.routes.ts`): all routes require `requireRole(['SUPER_ADMIN'])`

If the admin user is role `ADMIN` (not `SUPER_ADMIN`), the sidebar hides the coupons section AND the backend rejects all coupon API calls.

**Fix:** If the intent is for `ADMIN` users to manage coupons:
1. Change `AdminSidebar.tsx` line 77: `coupons: "SUPER_ADMIN"` → `coupons: "ADMIN"`
2. Change `coupon.routes.ts` role requirements from `['SUPER_ADMIN']` to `['ADMIN']`

### Fix 6: Banners not saving (create/update fails)

**Root cause:** snake_case vs camelCase field name mismatch:

| Frontend sends | Backend expects | Outcome |
|---|---|---|
| `image_url` | `imageUrl` | Unknown field, dropped |
| `link_url` | `linkUrl` | Unknown field, dropped |
| `sort_order` | `sortOrder` | Unknown field, dropped |
| `is_active: 1` (number) | Not in create schema | Unknown field, dropped |

Additionally, `is_active: 1` is a number but the update schema expects `isActive: boolean`.

**Fix in `src/components/admin/banners/BannersView.tsx`:**
Change the create payload (around lines 112-120) from snake_case to camelCase:
```typescript
const payload = {
  title: formData.title,
  imageUrl: formData.image_url,        // was: image_url
  linkUrl: formData.link_url || undefined, // was: link_url
  position: formData.position,
  sortOrder: formData.sort_order,       // was: sort_order
  // is_active: 1,  // REMOVE — not in create schema, backend defaults to active
};
```

For update, use the `updateBannerSchema` fields which does accept `isActive` (boolean):
```typescript
const payload = {
  ...existingBannerData,
  isActive: formData.is_active === 1 || formData.is_active === true, // convert to boolean
};
```

## Phase 3: Admin Panel Core Fixes

**Goal:** Fix structural issues in the admin panel — snake_case/camelCase alignment and all field mismatches.

> **NOTE:** Blog is intentionally NOT managed from the admin panel. Blog posts stay hardcoded on the storefront (good for SEO). Do not add any blog admin module, sidebar entry, or store mapping.

### Background

The admin panel has a systemic problem: frontend components send snake_case field names (like `image_url`, `link_url`, `created_at`) but most backend Zod schemas expect camelCase (like `imageUrl`, `linkUrl`, `createdAt`). This is because the frontend was written against one API convention and the backend uses another. The database itself uses snake_case, but the API layer (Hono controllers + Zod schemas) uses camelCase in most places.

### Task 1: Audit and fix all snake_case/camelCase mismatches

For EACH admin module, check the frontend component payload against the backend Zod schema:

1. **Banners** (already identified in Phase 1 Fix 6 — confirm the fix is complete)
2. **Popups** — `src/components/admin/popups/PopupsView.tsx` vs `backend/src/modules/popups/popup.schema.ts`
3. **Gallery** — `src/components/admin/gallery/GalleryView.tsx` vs `backend/src/modules/gallery/gallery.schema.ts`
4. **Team** — `src/components/admin/team/TeamView.tsx` vs `backend/src/modules/team/team.schema.ts`
5. **Products** — `src/components/admin/products/ProductForm.tsx` vs `backend/src/modules/products/product.schema.ts`

> **Blog is excluded from this audit.** Blog stays hardcoded on the storefront and is NOT managed from admin.

For each pair:
1. Read the backend Zod schema to find the EXACT expected field names
2. Read the frontend component to see what field names it sends
3. Align the frontend to send what the backend expects (change frontend, not backend — unless the backend uses an inconsistent naming convention)
4. Also check `src/lib/api/admin.ts` — the API client functions sometimes transform payloads

### Task 2: Fix notification component if still broken

**Read `src/components/admin/AdminNotifications.tsx`** and verify it uses the correct backend field names:
- `id` (not `_id`)
- `is_read` (not `isRead`)
- `created_at` (not `_creationTime`)
- Proper Lucide icons (not `"??"` strings)

If the component still uses Convex-era field names (`_id`, `isRead`, `_creationTime`), fix it to use the actual backend fields. Reference `backend/src/modules/notifications/` for the column names.
